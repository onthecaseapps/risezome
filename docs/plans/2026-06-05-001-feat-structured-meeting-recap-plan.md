# feat: Structured Meeting Recap (review page redesign)

## Summary

Replace the markdown-blob meeting recap with a **structured, KMS-encrypted JSON recap** — overview, timestamped key topics, categorized decisions, action items (text + assignee + timestamp), a participants list, and speaker count — generated by Claude on meeting completion, and redesign the review page to the provided mockup (stat cards, topic list, decision cards, display-only action checklist, participants rail, Regenerate). Display-only throughout (no done-state/checkboxes). Backward-compatible: meetings that only have the legacy markdown recap still render.

Mockup: Claude Design bundle "Meeting Recap.html" (https://claude.ai/design/p/1eec42a5-69f0-4c30-b197-994f35b335c6?file=Meeting+Recap.html) — fetch at build time for exact markup/spacing/colors.

---

## Problem Frame

The post-meeting recap (`apps/portal/src/inngest/lib/meeting-recap.ts`, fired on completion via the `risezome/meeting.recap-requested` Inngest event) prompts Claude for a **prose markdown** blob (Overview / Key topics / Decisions / Action items as bullets), stored encrypted in `meetings.recap_text_enc` and rendered by a bespoke markdown-lite renderer on the review page. The mockup calls for a far richer, scannable surface: per-topic timestamps, decisions as categorized cards, action items with an owner + timestamp, a participants panel, stat counts, and a Regenerate control — none of which exist as structured data today.

This is a backend + frontend change: the recap must become **structured JSON** (model returns a typed object via Anthropic tool-use, not prose), stored in a new encrypted column, with participants/speaker-count derived from transcript speaker labels; then the review page renders the structured shape, degrading to the old markdown for pre-existing meetings.

---

## Key Technical Decisions

**KTD1. Structured recap via Anthropic forced tool-use, not prose.** Mirror the only structured-JSON pattern in the codebase — the relevance classifier (`packages/engine/src/relevance/anthropic-classifier.ts` + `prompt.ts buildRelevanceTool`): a single non-streaming `/v1/messages` call with `tools: [recapTool]` + `tool_choice: { type:'tool', name }`, an `input_schema` (JSON Schema) for `{ overview, topics[], decisions[], action_items[] }`, then scan `content` for the `tool_use` block and **hand-validate** `block.input` (no schema library, matching the classifier). Raise `max_tokens` well above the current 700 (sized for a short markdown blob; a richer JSON object with timestamped topics + action items will truncate) — target ~2000. Keep `temperature: 0`, the `cache_control: ephemeral` transcript block, and `claude-haiku-4-5`.

**KTD2. The model produces the narrative fields; participants + speakerCount are DERIVED, not model-invented.** The model returns `overview`, `topics`, `decisions`, `action_items` (it can cite timestamps and owner names because the transcript is fed with them — see KTD3). `participants` and `speakerCount` are computed from the meeting's distinct transcript speaker labels (`meeting_events.payload->>'speaker'`) — the same source `capture_card_stats` already aggregates — NOT asked of the model. This is the participant crux: named attendees exist only when Recall supplied `participant.name` (Zoom/Meet bot meetings); for **local-audio meetings `speaker` is null**, so `participants: []` / `speakerCount: 0` is the correct degraded state. There is no profiles table, `meeting_participants` is user_id-only, and `calendar_events` stores only `attendee_count` — so speaker labels are the only naming source.

**KTD3. Feed a TIMESTAMPED transcript to the recap model.** `transcriptWithText(db, meetingId, orgId)` returns per-utterance `payload.startMs` + `payload.speaker` + decrypted `text`. Today `generate-meeting-recap.ts` and `buildTranscriptText` flatten to `${speaker}: ${text}` and drop `startMs`. Change the flatten + the transcript-text builder to include `[mm:ss]` per line so the model can attach a `timestampMs` to each topic/action. Timestamps are **display-only** in v1 (no deep-link/jump-to a transcript view).

**KTD4. Store the structured recap in a NEW encrypted column `recap_json_enc` (+ `recap_json_key_version`).** Mirror the `recap_text_enc` pattern: a nullable `bytea` column holding `encryptForOrgToBytea(orgId, JSON.stringify(recap))`, stamped `recap_json_key_version = CRYPTO_VERSION.KMS_ESDK`, read via `decryptForOrgFromBytea` + `JSON.parse`. A new column (rather than overloading `recap_text_enc`) keeps backward-compat trivial: old meetings have `recap_text_enc` (markdown) and `recap_json_enc = null`; new meetings have `recap_json_enc`. **Register the new column in `apps/portal/src/inngest/lib/encrypted-columns.ts` (`ENCRYPTED_COLUMNS`)** — the KMS backfill + per-org key rotation walk that list in lockstep; an unregistered encrypted column is silently skipped and a rotated org can't decrypt its recap. (see docs/solutions/2026-06-03-content-encryption-at-rest.md)

**KTD5. Display-only ⇒ no new RLS, no client writes.** The recap rides on `meetings`, which has a participant-scoped SELECT policy (`is_meeting_participant`, attendees ∪ super-admin — the privacy-ladder in the older roles doc was superseded by `20260609030000_attendees_only_access.sql`). A new column inherits that row policy — no new policy needed. All recap writes stay service-role (the Inngest function). Preserve the two app-layer invariants: the review page audits master-key reads (`recordMasterKeyAccessIfNeeded` in `app/_lib/meeting-access.ts`), and the captures list **excludes** master-key-only meetings so it never silently decrypts a restricted recap without an audit. (see docs/solutions/2026-06-04-roles-and-meeting-privacy.md)

**KTD6. Backward compatibility: prefer JSON, fall back to markdown.** The redesigned review page reads both columns. When `recap_json_enc` decrypts to a valid structured recap → render the new layout. When only `recap_text_enc` exists (old meetings) → render the existing `RecapMarkdown` in a compact section. Decrypt failures degrade to a null/empty state (mirror the live page's `EnvelopeCryptoError → null`, never 500). The captures-list preview currently derives its summary line from the markdown string (`firstLine`) — switch it to read `overview` from the JSON when present, else the markdown first line.

**KTD7. Decision category is a model-assigned short free-form tag.** The mockup's decision cards carry a one-word category (Commercial, Ventel, Schema…). The `input_schema` includes `decisions[].category: string` with prompt guidance to emit a short (≤2-word) topical tag; not a constrained enum (the categories are meeting-specific, e.g. product names).

## High-Level Technical Design

Generation (Inngest, on `risezome/meeting.recap-requested`) and read (review page) data flow:

```
meeting completed
  └─ inngest: generate-meeting-recap  (concurrency 1/meetingId, retries 3)
       ├─ load transcript: transcriptWithText -> rows{ startMs, speaker, text }
       ├─ recap_status = 'generating'
       ├─ derive participants[] + speakerCount  (distinct payload.speaker; [] for local-audio)
       ├─ recapMeetingStructured(timestamped transcript) -> tool_use JSON
       │     { overview, topics[{text,timestampMs}], decisions[{category,text}],
       │       action_items[{text, assignee?, timestampMs}] }
       ├─ merge derived participants/speakerCount into the recap object
       ├─ encrypt: recap_json_enc = encryptForOrgToBytea(JSON.stringify(recap))
       │           recap_json_key_version = KMS_ESDK
       └─ recap_status = 'done'        (onFailure -> 'failed'; no-transcript -> done placeholder)

review page (server component, user RLS client)
  ├─ select recap_json_enc, recap_text_enc, recap_status   (+ master-key audit)
  ├─ recap_json_enc present? decrypt + JSON.parse -> StructuredRecap   (degrade null on EnvelopeCryptoError)
  │     -> <RecapClient structured>  stat cards | overview | topics | decisions | action items | participants rail | Regenerate
  └─ else recap_text_enc -> markdown fallback (existing RecapMarkdown)
```

The structured recap shape (directional):

```
StructuredRecap {
  overview: string
  topics:        { text: string; timestampMs: number | null }[]
  decisions:     { category: string; text: string }[]
  action_items:  { text: string; assignee: string | null; timestampMs: number | null }[]
  participants:  { name: string }[]      // derived from speaker labels; [] when none named
  speakerCount:  number
}
```

## Scope Boundaries

- Out: the live in-meeting page, the retrieval/synthesis pipeline, the knowledge-gaps surface.
- Out: interactive action items — **no checkboxes / done-state / persistence / writable RLS** (display-only, confirmed with the user).
- Out: transcript jump-to / deep-linking from a timestamp (timestamps are display-only in v1).
- Out: resolving real attendee identities beyond transcript speaker labels (no calendar-attendee parsing, no Recall participant-roster persistence, no profiles table) — `participants` is speaker-derived, empty for local-audio.

- Timestamp deep-link into a transcript view.

## System-Wide Impact

- **Encryption registry:** the new `recap_json_enc` MUST be registered in `ENCRYPTED_COLUMNS` (KTD4) or org key rotation/KMS backfill silently skips it.
- **Two read surfaces** consume the recap: the review page (full structured) and the captures list (overview preview) — both must handle the JSON-or-markdown duality + decrypt degrade + master-key invariants.
- **No migration backfill / no breaking change:** old meetings keep `recap_text_enc`; the column add is additive and nullable.
- **Cost:** the recap call's `max_tokens` rises (~700 → ~2000) and the transcript now carries timestamps — a modestly larger single Haiku call per meeting completion (and per Regenerate).

---

## Risks & Mitigations

- **Model returns prose instead of tool_use / malformed JSON.** Mitigation: forced `tool_choice` makes tool_use near-certain; hand-validation coerces/skips bad items and throws only on a wholly-unusable response (→ `recap_status='failed'`, which the page renders).
- **Token truncation** on long meetings. Mitigation: raised `max_tokens`; the transcript is already cache_control-blocked. If still truncated, the parse degrades per-item.
- **Participant emptiness surprises** (local-audio shows no attendees). Mitigation: this is the accepted, documented fallback; the rail collapses and the stat shows 0 — explicitly tested (U3, U4).
- **Backward-compat regressions** on old meetings / decrypt blips. Mitigation: JSON-or-markdown fallback + `EnvelopeCryptoError → null` degrade, tested in U4/U5.
- **Captures preview break** (markdown parser on JSON). Mitigation: U5 reads `overview` from JSON.
