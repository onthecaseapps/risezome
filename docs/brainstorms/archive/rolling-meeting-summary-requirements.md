# Rolling Meeting Summary

## Problem Frame

The just-shipped recent-context fix (passing the last 5 finalized
utterances + heuristic fragment merge to the synthesizer) handles
short-range context — questions whose preceding turns are within the
last 60 seconds. Live testing surfaced the longer-range gap: by
minute 20 of a meeting, the user can ask "about that auth flow we
were discussing" and the system has no idea what that means. The
5-utterance window can't reach back that far; sending the whole
transcript per synthesis call would balloon input tokens linearly
with meeting length and burn cost.

Additionally — and this is the more subtle problem — the relevance
classifier currently judges each utterance in isolation. "in the
app" alone looks like filler regardless of what was said before.
With prior context ("are any LLMs leveraged"), the same utterance is
a coherent continuation worth synthesizing. The classifier needs to
know what's been discussed to make sense of what it's classifying.

A rolling meeting-level summary, generated by a side-call to Claude
on a pause-debounced cadence, gives both consumers the long-range
context they need without paying the linear-token cost on every
retrieval call.

### What this is NOT

This is **not** a user-facing "meeting summary" feature. The prior
synthesis-card brainstorm (`docs/brainstorms/llm-synthesis-card-requirements.md`)
and the live-page synthesis-first brainstorm
(`docs/brainstorms/live-page-synthesis-first-requirements.md`) both
explicitly list "summary-of-the-meeting" under **Outside this
product's identity** — drifting toward generic AI-meeting-assistant
positioning. This brainstorm respects that constraint: the rolling
summary is **internal infrastructure**, never rendered to the user.
It exists as input to better retrieval/synthesis/classification, not
as output.

## Decisions

### D1. Cadence: pause-debounced, count-or-time triggered

The rolling summarizer fires when **either** N final utterances **or**
M seconds have accumulated since the last summary, **and** the most
recent utterance is at least D seconds old (the pause-debounce — avoid
summarizing mid-thought).

Defaults to start with:
- N = 15 final utterances
- M = 120 seconds
- D = 10 seconds talking-pause

Adjust based on live-test cost + freshness signal. The debounce is
the load-bearing decision: clock-only refresh that fires while
someone is still talking produces stale summaries that miss the
current thread.

*Considered and rejected:*
- **Fixed 2-minute interval** — refreshes mid-sentence, no debounce.
- **Per-utterance regeneration** — too expensive; ~$0.30 per minute
  of meeting and the per-call payload is mostly stable.
- **Topic-shift-only triggering** — would need a separate detector;
  pause + count is a simpler approximation.

### D2. Summary shape: hybrid prose + structured fields

Each summary is a JSON object emitted by Claude:

    {
      "summary": "Free-form prose paragraph (1-3 sentences) describing what the meeting has covered so far.",
      "current_topic": "Short label of the active thread (e.g., 'Auth flow error handling').",
      "open_questions": ["Verbatim questions raised but not yet resolved."],
      "key_terms": ["Project-specific nouns: filenames, identifiers, plan U-IDs mentioned"]
    }

- `summary` feeds the synthesizer as long-range prose context
- `current_topic` and `open_questions` feed the relevance classifier
  so it can judge "does this utterance make sense as a continuation
  of one of these threads?"
- `key_terms` are an optional retrieval boost — concatenated with the
  utterance query for embedding so the rolling window of vocabulary
  follows the conversation

*Considered and rejected:*
- **Prose-only** — too vague for the classifier to do useful
  reasoning over; rebuilds as a structured form within a release.
- **Free-form JSON-mode** — Claude reliably emits structured output
  when given a clear schema in the prompt; locking the shape avoids
  downstream field-presence guards.

### D3. Synthesizer integration: summary prepended to recentContext

The existing `SynthesisInput.recentContext` field (added in the
preceding commit) becomes the carrier for long-range context too.
The bot-worker constructs `recentContext` as:

    [
      "<summary.summary prose>",     // single entry, marked as overview
      ...last 5 final utterances     // unchanged from current behavior
    ]

The synthesizer's `buildUserMessage` already labels these as
"Recent transcript (oldest first; treat the most recent utterance as
the question)" — the prose overview slots in as the oldest entry,
giving Claude a long-range anchor before the recent turns.

Alternative considered: a dedicated `meetingContext` field separate
from `recentContext`. Rejected for V1 — the existing field handles
both shapes naturally and avoids prompt-template churn.

### D4. Relevance classifier integration: summary as classifier context

The existing `AnthropicRelevanceClassifier` already makes a Claude
call on ambiguous utterances. Its prompt grows to include:

- The latest `current_topic` from the rolling summary
- The latest `open_questions` list
- The current utterance to classify

This reframes the classifier's job from "is this utterance
substantive in isolation?" to "given what's been discussed, does
this utterance make sense to pass along for retrieval?" The same
LLM call cost; better-grounded judgment.

The classifier still outputs the existing three-state result
(`clearly_filler` / `clearly_substantive` / `ambiguous`). No
breaking change to downstream gating.

*Considered and rejected:*
- **Embedding similarity** to `current_topic` as a deterministic
  gate. The user clarified the goal isn't off-topic filtering but
  coherence-in-context judgment — embedding similarity is the wrong
  shape for that. Keeping LLM judgment with better input.

### D5. Storage: in-memory only for V1

The full transcript and the most-recent summary live in the
bot-worker runtime state for the duration of the meeting. On meeting
end (or WS disconnect with timeout), the runtime is dropped and the
transcript + summary go with it.

No DB persistence in V1. The post-meeting summary (already in scope
elsewhere — `R15` in the meeting-context-copilot brainstorm) is a
distinct artifact that can be produced from `meeting_events` if
needed; the rolling summary is operational, not durable.

Privacy: respects the existing trust model — the transcript already
flows through Deepgram and into `meeting_events` rows; the in-memory
rolling summary doesn't expand the data surface. The summary call
shares the existing Anthropic consent grant.

### D6. Cost safeguards

Hard cap: maximum 1 summary per 60s wall-clock, regardless of
trigger logic. Bounds the worst case if the pause-debounce keeps
firing right at the boundary.

Bounded transcript size: if the in-memory transcript exceeds
`MAX_TRANSCRIPT_CHARS` (default 20K chars, ~5K tokens), drop the
oldest utterances before sending to Claude. Long meetings get a
sliding-window view of recent N minutes for summarization; the
summary itself carries the longer-range memory in compressed form.

Refusal handling: if Claude refuses to summarize (malformed
transcript, prompt edge case), retain the prior summary unchanged.
Don't fail open to no-summary — the just-shipped recent-context fix
remains as a baseline.

## Defaults

- **First summary doesn't fire until ~M seconds of meeting** —
  pre-summary period falls back to the existing recentContext
  behavior (last 5 finals only). Gracefully degrades on cold start.
- **Summary updates atomically** — synthesizer + classifier always
  see the same summary version. No partial-update race.
- **Synthesis can still run without a summary** — the new context is
  additive, not gating.
- **Same Anthropic key + model** as the existing synthesizer +
  classifier. No new consent surface.
- **Telemetry**: log every summary call with input transcript char
  count, output tokens, latency, refusal flag. Same pattern as the
  existing `synthesis.done` log line.

## Scope Boundaries

### In scope
- New "MeetingSummarizer" component in the bot-worker runtime
- Rolling buffer of finalized utterances (capped by char count)
- Pause-debounced trigger that fires the Claude summary call
- Hybrid prose+structured output schema
- Wiring summary into `SynthesisInput.recentContext`
- Wiring summary into relevance classifier prompt
- Cost cap + refusal handling
- Local-mic debug page (`/debug/live-mic`) shows current summary in a
  panel for prompt iteration visibility

### Deferred for later
- **Cross-meeting memory.** Summaries from prior meetings as
  retrievable context for future meetings. Approach 3 from the
  brainstorm — worth revisiting if single-meeting V1 proves the
  pattern.
- **User-facing summary surface.** No "current meeting context"
  sidebar in the live page UI. The summary is internal-only by
  default; the debug page exposure is the only render.
- **Embedding-similarity gating.** Deterministic on-topic check
  alongside the LLM classifier. The user clarified the goal is
  context-aware coherence, not thematic filtering.
- **DB persistence of rolling summaries.** In-memory only; later
  features can add `meeting_summaries` table if needed.
- **Per-speaker summaries.** Single meeting-level summary; speaker-
  attributed summaries are a future refinement.
- **Adaptive cadence tuning** — fixed N/M/D defaults; instrument
  cost + freshness signals before adding feedback control.

### Outside this product's identity
- **User-facing meeting summary as the headline feature.** Drifts
  into AI-meeting-assistant territory. The rolling summary stays
  internal. Post-meeting summary (already deferred elsewhere) is a
  different artifact and a different decision.

## Dependencies and Assumptions

- **Anthropic API** continues to be the synthesizer + classifier +
  summarizer provider. Same key, same consent grant. The summarizer
  call is a NEW use of the key — verify the consent grant covers
  general synthesis-flavored use, or add `summarizer` as a recognized
  scope if the consent module enumerates uses.
- **Recent-context plumbing landed** — the just-shipped
  `SynthesisInput.recentContext` + `buildUserMessage` change is the
  carrier path. No new prompt-template work for the synthesizer side.
- **Relevance classifier prompt** has room to grow without busting
  the cache or context window. Worth checking the size budget
  against the existing few-shot prefix.
- **Meeting duration ceiling**: 4 hours assumed maximum (Recall bot
  + WS timeout combination). MAX_TRANSCRIPT_CHARS sizing should keep
  Claude calls below the 200K context window even at the ceiling.

## Success Criteria

1. **Coherence-in-context test**: speaker says "are any LLMs
   leveraged" → pauses → "in the app and where in the code base are
   they." The second utterance is classified as `clearly_substantive`
   (not filler), retrieval fires, and synthesis answers coherently
   referencing LLM usage in the codebase.
2. **Long-range memory test**: 20 minutes into a meeting that has
   discussed auth flow at minute 5, the speaker says "about that auth
   flow we were discussing." Retrieval surfaces auth-related sources
   (the summary's `current_topic` or `key_terms` carry the auth
   context through to the embedding query).
3. **Side-chat coherence test**: speaker says "uh… so… what time is
   lunch?" — classifier returns `clearly_filler`. Speaker says
   "actually let me check the deploy status" in a meeting that's been
   discussing deploys — classifier returns `clearly_substantive`
   (same utterance shape, different context, different judgment).
4. **Cost budget**: rolling summary calls add <$1/hour of meeting
   duration to Anthropic spend in a typical 60-minute meeting (15-20
   summary calls × ~$0.01-0.02 each).
5. **Latency budget**: summary call completes in <2s p95. Doesn't
   block any synthesis call — the synthesizer reads whatever summary
   is currently in runtime state, even if a refresh is in flight.
6. **Cold-start grace**: first 2 minutes of a meeting work as well
   as today (no regression on questions asked before the first
   summary lands).
7. **Debug page reveals the current summary**: developers can iterate
   on the summarizer prompt by watching the summary update in real
   time on `/debug/live-mic`.

## Open Questions (deferred to planning)

- **Exact summarizer prompt shape**. Few-shot examples, refusal
  conditions, schema-enforcement strategy (Anthropic's JSON mode,
  prompt-only structure, or post-parse validation). Planning needs
  to pick.
- **Whether `key_terms` should expand the EMBEDDING query** (boost
  retrieval) **OR only the synthesis prompt** (boost answer
  grounding). Probably both, but instrument before committing.
- **How summary updates interact with in-flight synthesis calls**.
  Today the synthesizer call captures `recentContext` at call time;
  if a new summary lands mid-stream, the in-flight call uses the
  prior summary (consistent). Whether this is right needs live-test
  observation.
- **Speaker attribution in the summary**. Today's prompt would
  produce a single-perspective summary. Multi-speaker context might
  need "Speaker A asked X; Speaker B responded Y" structure. Defer
  until a real multi-speaker test surfaces the gap.
